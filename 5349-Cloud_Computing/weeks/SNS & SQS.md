## 错误设计（强耦合）:
User → EC2(App) → 依次做所有事情
                 → 生成缩略图
                 → 压缩图片
                 → 发邮件
问题：
- 用户要一直等（同步）
- 某一步慢 → 整个系统慢
- 某一步挂 → 全部失败
- 无法扩展
## 引入 SQS + SNS
### SQS 是“任务队列”： 把任务丢进去，让别人慢慢做
User 上传图片
        ↓
EC2（Producer）
        ↓
把任务丢进 SQS Queue
        ↓
Worker服务器（Consumer）
        ↓
慢慢处理（生成缩略图）
### SNS 是“广播系统”： 一条消息 → 通知很多人
SNS + SQS

|SNS |	SQS|
|----|----|
|广播 |	缓冲|
|push |	pull|
|不存储 | 存储|

### 例子：电商下单
用户下单后：

- SNS 做什么？
通知：
库存系统
支付系统
发货系统
- SQS 做什么？
每个系统内部：
排队处理订单
防止系统崩

###
什么时候用 SQS？👉 需要异步处理任务
什么时候用 SNS？👉 需要广播消息
为什么要解耦？👉提高扩展性，提高稳定性，防止级联崩溃
